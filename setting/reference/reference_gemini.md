1. 実証研究：AIは本当に速いのか？（定量的評価）
個人のタスク完遂時間を計測したランダム化比較試験（RCT）の結果です。ツール提供側の主張と、中立的な研究結果の対比が鮮明になっています。

MIT Economics (Demirer et al., 2024): 「The Effects of Generative AI on High-Skilled Work」

内容: 4,800名以上の開発者を対象とした大規模なフィールド実験。

結論: AI利用によりタスク完了数が26.08%増加。特に「スキルの低い層」や「経験の浅い層」で最大40%近い向上が見られた一方、シニア層への恩恵は限定的（7〜16%）であった。

METR (2025): 「Measuring the Impact of Early-2025 AI on Experienced Open-Source Developer Productivity」

内容: ベテランのオープンソース開発者を対象にした最新の調査。

結論: 驚くべきことに、AI（Claude 3.5/Cursor等）を使用したグループは、使用しなかったグループよりもタスク完了に19%長くかかった。

示唆: 「AIが速い」という主観的な感覚（開発者は20%速くなったと誤認していた）と、実際のパフォーマンスの間に**乖離（認知の歪み）**があることを警告しています。

2. 構造的課題：コード品質と「負の遺産」（メンテナンス性）
AIが「書きすぎる」ことによる弊害をデータで示したレポートです。

GitClear (2024-2025): 「Coding on Copilot: 2024 Data Report」

内容: 1.5億行のコード変更データを分析。

結論: AI導入後、コードのChurn（書いてすぐに消されるコード）が倍増し、既存コードの「再利用」や「リファクタリング」が減少、「コピペによる追加」が急増している。

リスク: 短期的なデリバリー速度は上がるが、中長期的な技術負債を劇的に加速させている可能性を指摘。

DORA (Google Cloud) 2024: 「State of DevOps Report」

内容: ソフトウェア開発の世界的指標であるDORAによる最新報告。

結論: AI採用によりコード生成速度は上がったが、それが**「バッチサイズの増大」**を招いている。

影響: 小さな変更を頻繁に出すというDevOpsの基本原則が崩れ、レビュー負荷の増大やデプロイの不安定化という副作用が出始めている。

3. 開発者体験（DevEx）と認知的負荷
生産性を「アウトプット量」ではなく「フロー状態」で捉える研究です。

Microsoft / GitHub (2025): 「It’s About Time: The Copilot Usage Report」

内容: 数千万件の会話ログと開発者行動の分析。

結論: AIは「書く」作業よりも「調べる・理解する」という認知的負荷の軽減に最も寄与している。

視点: 生産性の真の向上は、コード行数ではなく「フロー状態（没入状態）を維持できる時間の長さ」に現れる（SPACEフレームワークのSatisfaction/Well-beingへの寄与）。

JetBrains (2026): 「AI Tool Switching Is Stealth Friction」

内容: 開発ツール内でのAI活用と文脈切り替えのコストに関するレポート。

結論: 複数のAIツールを使い分けることによる「ツール・スイッチング」が、新たな摩擦（ステルス・フリクション）となり、集中力を削いでいる現状を報告。

4. 組織と戦略：持続可能な優位性
AIを組織としてどう位置づけるべきかという経営・戦略的視点です。

Qodo (2025): 「State of AI Code Quality in 2025」

内容: AIによる品質担保（テスト自動生成やレビュー）に成功しているチームの分析。

結論: 単にコードを書かせるだけでなく、**「AIによるチェック」**のプロセスを組み込んでいるチームのみが、速度と品質のトレードオフを克服している。

McKinsey (2024): 「Yes, you can measure developer productivity」の再考

内容: かつて物議を醸した「個人の生産性測定」に関する議論のAI時代版。

結論: AI時代には個人のタスク完了速度はもはや指標にならず、**「AIが生成した複雑なシステムをいかに統制できているか」**という組織のガバナンス能力が真の生産性を分かつ。
