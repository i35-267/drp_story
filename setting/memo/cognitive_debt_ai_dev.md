 # 認知的負債（Cognitive Debt）とAI時代の開発組織

## 1. 背景：コード生成速度と理解速度の乖離

- **AIによる開発支援の前提変化**  
  - 従来：コードを書く行為そのものが、設計・実装の理解を強制していた（生産＝理解がほぼ連動）。  
  - 現在：プロンプトから一気にコードが生成され、エンジニアはレビューと微修正に集中する構図に変化。
- **「理解を飛ばす」ことで上がる速度**  
  - 実装の細部を頭に保持しなくても、動くコードとテストを素早く手に入れられる。  
  - その結果、**短期的な開発速度は大きく向上**する。
- **しかし理解速度は人間の処理能力に依存**  
  - 出力は一桁速くなっても、「なぜこう動くのか」を本当に理解する速度は、ほとんど変わらない。  
  - この「出力速度」と「理解速度」のギャップが、認知的負債の温床になる。

## 2. 認知的負債の定義と技術的負債との違い

- **技術的負債**  
  - 意図的／非意図的な設計上の妥協により、将来的な修正コスト（利息）が増大する状態。  
  - コードの複雑さ、重複、アーキテクチャの歪みなど、**コードやシステムそのものに存在する負債**。
- **認知的負債（Cognitive Debt）**  
  - コードやシステムが「なぜそのように動いているか」「なぜその設計になったか」を、組織として／個人として**理解できていない状態**。  
  - システムは動くしテストも通るが、**振る舞いと設計意図を説明できない**。
- **違いのポイント**  
  - 技術的負債：ソフトウェアに埋まっている欠陥・妥協。  
  - 認知的負債：**人間側の理解の欠如**（理解がついてきていないこと自体が負債）。  
  - 技術的負債は静的解析やメトリクスで一部可視化できるが、認知的負債は**現状ほぼ可視化されない**。

## 3. 認知的負債が生まれるメカニズム

- **生産と理解の分離（Comprehension Lag）**  
  - AIが数百行のコードを一気に生成し、エンジニアは「ざっと読んで動かす」だけで機能が完成する。  
  - 出力速度は上がる一方で、理解のための時間は圧縮されるか、そもそも確保されない。  
  - 結果として、「書いた本人ですら、自分のコードを初見の他人のコードのように感じる」状態が生まれる。
- **レビュアーのジレンマ**  
  - かつて：ジュニアの出力速度がボトルネックで、シニアはそれより速くレビューできた。  
  - 今：AIとジュニアのコンビが大量のコードを吐き出し、**シニアのレビュー速度がボトルネック**になる。  
  - 選択肢は「レビューの質を維持してボトルネックになる」か「レビューを浅くしてスループットを優先する」か。現実には後者に傾きやすい。  
  - その結果、「**レビュアーもよく分かっていないコードに承認スタンプが押される**」ことで、認知的負債が組織的に増幅する。
- **組織的記憶と人材育成への影響**  
  - 組織の知識は、ドキュメントなどの**明示知**と、実装や運用を通じて身につく**暗黙知**で構成される。  
  - AIによって「理解を飛ばしたままでも変更ができる」ようになると、暗黙知の形成が遅れる／起きない。  
  - 特にジュニア層が、苦労しながら実装することで得ていた「設計判断の勘所」や「痛みを通じた学習」を積みにくくなり、  
    長期的には**Staffクラスのアーキテクト候補が育ちにくい組織**になる。

## 4. 表面化のタイミングと失敗モード

- **技術的負債よりも「後」に、「大きく」表面化する**  
  - 認知的負債は、コードが動いている限りは表面化しない。  
  - 大きな設計変更、予期しない障害、新しい要件への対応など、「理解」が必要になるタイミングで一気に顕在化する。
- **典型的な3つの失敗モード**
  - **(1) 古いコードほど危険になる**  
    - 普通は「長く動いているコードほど信頼できる」と考えるが、  
      AI時代のコードは「**ほとんど理解されないまま放置され続けたコードほどブラックボックス化が進む**」。  
  - **(2) インシデント時のブラックボックス化**  
    - 深夜のアラート対応で、オンコールが「自分も組織もよく理解していないAI生成コード」を相手にする。  
    - 本来10分で直せたはずの不具合が、4時間の発掘作業になる。  
    - この遅延が、MTTR（平均復旧時間）やCFR（変更失敗率）の悪化として、数ヶ月後に統計上現れる。  
  - **(3) 将来のシニア層の枯渇**  
    - ジュニアがAIの補助なしでは実装したことがないまま年次を重ねると、  
      自分で設計し、責任を持ってシステム全体を引き受ける経験を積みにくい。  
    - 組織は「**5年後のStaff Engineer候補を、今期の速度向上と引き換えに失っている**」かもしれないが、それは統計にも人事KPIにも出てこない。

## 5. 測定されないものを最適化している問題

- **既存のメトリクスは「理解されていること」を暗黙に前提**  
  - ストーリーポイント消化数、マージ数、リードタイム、DORA指標など、  
    いずれも「実装した人・レビューした人は、そのコードを理解している」という前提で設計されてきた。  
  - 生産と理解が連動していた時代には、この前提は概ね妥当だった。
- **今は「表面的な理解」でも機能を出荷できる**  
  - AIにより、実装の細部を深く理解していなくても、動く機能を高速にリリースできてしまう。  
  - つまり、「機能が出荷された＝理解されている」という歴史的な前提は、もはや成り立たない。
- **認知的負債は客観指標でほぼ測れない**  
  - 「自分の書いたコードを再読なしに説明できるか」「システムの振る舞いを前もって予測できるか」  
    といった理解度は、現状のパフォーマンス指標やダッシュボードには載らない。  
  - その一方で、**短期的なスピード指標は非常に分かりやすく、組織内で強いインセンティブを持つ**。
- **結果として起きる力学**  
  - 「理解に時間をかける人」よりも、「理解を飛ばしてでも機能を高速に出す人」のほうが  
    数字上は優秀に見えやすい。  
  - 評価・昇進・予算配分は、測定可能な速度指標に引っ張られ、  
    **認知的負債を加速させる行動を選好するように組織が最適化されてしまう**。

## 6. 適用範囲の限界と緩和の方向性

- **すべての仕事に深い理解が必要なわけではない**  
  - 一時的なスクリプトや、影響範囲の小さい実験的機能など、  
    「理解よりもスピードが優先される」領域は確かに存在する。  
  - 認知的負債の議論は、**アーキテクチャ判断や長期保守が前提の領域**に特に強く当てはまる。
- **実際の状況は「もともと理解されていなかった」部分もある**  
  - 過去から現在にかけて、すべてのコードが十分理解されていたわけではない。  
  - ただし、AIによってその「理解不足の割合」と「速度」が大きく増幅している可能性がある。
- **緩和のための方向性の例**  
  - どのレイヤー・どの機能に対しては「深い理解」を要求するのかを、組織として明示する。  
  - 設計レビューやアーキテクチャレビューの場で、「実装方針」だけでなく  
    「**なぜその構造なのか**」「どのような代替案を捨てたのか」を言語化・記録する。  
  - ペアプロ／モブプロ、設計ドキュメントの強化、AI利用ポリシーの整備などを通じて、  
    「理解の形成」にも時間と評価を割く仕組みを持つ。

## 7. まとめ：測れないものをどう扱うか

- 認知的負債とは、**コードやシステムの挙動・設計意図に対する理解の欠如が、静かに累積していく状態**を指す。  
- AIによって開発速度は上がる一方、その影で「理解を飛ばす」行為が常態化すると、  
  技術的負債よりも後になって、より大きなコストとして表面化する危険がある。  
- 現在の評価・計測システムは、この負債をほとんど捉えられておらず、  
  **短期的なスピードばかりを最適化する力学**が強く働きうる。  
- 組織としては、「どこで速度を取り、どこで理解を優先するか」を意識的に線引きし、  
  測りにくい「理解」そのものを、少なくとも議論と投資の対象に引き上げる必要がある。
