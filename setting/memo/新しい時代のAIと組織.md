## ブログの要点一覧

| テーマ | 内容 |
|--------|------|
| スループットの天井 | rate limit・理論値（daily 300 commit / 60,000行等）。制約要因は各サービスのrate limit |
| プロジェクトの硬さ | 柔らかい（グリーンフィールド、要件明確、タスク分離、自動検証、自己判断）vs 硬い（既存改修、要件曖昧、直列依存、目視確認、レビュー待ち、外部依存）。Agentに振って速く回せるかは硬さで決まる |
| 人間がボトルネック | 指示待ち・検収待ち・対応荒れ（特に厄介）。「考える仕事」の時間を確保する設計が必要 |
| ロードタイム | プロジェクト想起・検収までにかかる時間。3秒と3分で生産性が桁違い。計測し短くする仕組みが必要 |
| チーム設計 | アイデアマン（1）・ランナー（1）・レビュアー群（3+）。ピザ1枚は1人。レビュアーを厚くする |
| 完全自動化のリスク | 文脈の脆さ、コンテキストウィンドウの有限性。人間の介入を最小限の認知コストで最大効果にする設計 |
