# コードレビュー終焉論（How to Kill the Code Review）

**出典**: [How to Kill the Code Review](https://www.latent.space/p/reviews-dead)（Latent.Space）  
**著者**: Ankit Jain（Aviator CEO）  
**日付**: 2026年3月2日

---

## 要約

AI活用が進んだチームでは、タスク完了が21%増、マージが98%増している一方で、PRレビュー時間は91%増加している（1万超の開発者・1,255チームのデータ）。変更の**数**も**規模**も指数的に増えており、人間がコードを全て読むことはもはや現実的でない。

著者の提案は、**レビューの対象をコードから意図（Intent）へ移す**ことである。人間がやるべきは500行のdiffを読むことではなく、仕様・制約・受け入れ基準を定義すること。コードは spec の成果物に過ぎない。

信頼はスイスチーズモデルで多層的に構築する：複数エージェントに競わせて最良を選ぶ、テスト・型チェック・契約検証などの決定的ガードレール、BDDで人間が受け入れ基準を定義し機械が検証、エージェントの権限を最小スコープに制限、書く側と検証する側を分離する敵対的検証。特に重要なのは「検証基準はコードより先に、spec から定義する」という点。エージェントにコードもテストも書かせると、問題が移動しただけになる。

「遅くレビューして、どうせバグを見逃して、本番でデバッグする」から「速く出して、すべてを観測して、もっと速く戻す」へ。人間は機械より多く読むのではなく、上流でより深く考える側に回る。

---

## 1. 問題認識（記事の前提）

人間がコードを書いていた時代でも、レビューは追いついていなかった。多くのエンジニアリング組織に共通する「 dirty secret 」として、PRが何日も滞留する、ゴム印的な承認、レビュアーが500行のdiffを流し読みする（自分たちの仕事もあるため）、といった実態がある。

コードレビューは品質ゲートだと語られるが、実際には何十年も line-by-line のレビューなしでリリースしてきたチームは存在する。コードレビューが一般的になったのは2012〜2014年頃であり、それを覚えているベテランは少ない、という指摘がある。

また、レビューをしていても不具合は起きる。そのため、レビューだけに頼らず失敗に備えた設計がされてきた。フィーチャーフラグ、段階的ロールアウト、即時ロールバックなどがその表れである。

AIが書くコードのレビューは「同僚のコードより労力がかかる」と開発者が感じており、生産量の増加とレビュー負荷の増加が同時に起きている。この戦いは手動のコードレビューでは勝てない。コードレビューは、仕事の形に合わなくなった歴史的な承認ゲートである、と著者は結論づける。

---

## 2. なぜ「AIによるコードレビュー」では不十分か

AIによるコードレビューは、時間を稼いでいるに過ぎない。AIがコードを書き、AIがレビューするなら、その結果を表示するためのレビューUIや、レビューサイクル間のCIリソース・バージョン管理を挟む必然性が薄れる。AIコードレビューの価値は、開発サイクルの中でより左（上流）にシフトしていく。

人間がコードを書き、第三者の目が必要だった時代には、PR後のレビューは意味があった。エージェントがコードを書く場合、「第三者の目」は同じ盲点を持つ別のエージェントになる。価値があるのは承認ゲートではなく、イテレーションループそのものだ。

エージェントは常に信頼できるわけではない。人間には「AIが一度愚かなことをしたのを見つけたから、いつもチェックしなければ」と思う傾向がある。手動検証が現実的だった時代にはその直感は有効だったが、現在のスケールではもう成り立たず、今後さらに悪化する。

---

## 3. コードのレビューから意図（Intent）のレビューへ

人間のチェックポイントを上流に移す。コードをレビューしない、という考えは怖く感じるかもしれないが、ソフトウェア開発では以前もチェックポイントは移動してきた（ウォーターフォールのサインオフから継続的インテグレーションへ）。同じように、もう一度移すことができる。

Spec-driven development（仕様駆動開発）が、AIと働く主要なやり方になりつつある。人間がレビューすべきなのは、仕様・プラン・制約・受け入れ基準であり、500行のdiffではない。

この新しいパラダイムでは、**Spec が唯一の真実の源（source of truth）** となる。コードは spec の成果物である。コードをレビューする必要はなく、ステップ・検証ルール・コードが満たすべき契約をレビューする。

人間が関与する承認は、「正しく書いたか？」から「正しい問題を、正しい制約で解いているか？」へ移る。最も価値のある人間の判断は、最初の1行が生成される**前**に行使される。

---

## 4. 信頼の多層化：スイスチーズモデル（5層）

「コードは人間が書いてはいけない」「コードは人間がレビューしてはいけない」というルールを、どこまで受け入れられるか。LLMは指示に従うのが苦手で、逸脱し、自己検証も信頼できない（コードが炎上していても「動く」と答える）。解決策はLLMに検証させることではなく、**検証するスクリプトを書かせる**こと。判断から成果物へとシフトする。

信頼は多層で構築する。スイスチーズモデル：単一のゲートでは全てを捉えられない。穴が重ならないまで、不完全なフィルタを重ねる。

### Layer 1: 複数案の比較

1体のエージェントに正解を出させるのではなく、3体に異なるやり方で試させ、最良の結果を選ぶ。競わせる。オプションのコストは、ソフトウェア工学の歴史で最も安い。

選択は手動でなくてもよい。検証ステップを最も通過したもの、diffが最も小さいもの、新規依存を導入していないものなどでランクできる。競争によって、単一の試行では得られないシグナルが得られる。

### Layer 2: 決定的ガードレール

仕事を検証する決定的な方法が必要である。テスト、型チェック、契約検証——意見ではなく事実を返すもの。

LLMに「これは動いたか？」と聞くのではなく、pass/fail の成果物を出す検証ステップを定義する。エージェントは失敗したテストと交渉できない。仕様を満たすか、満たさないかだけである。

ガードレールはさらに層として定義できる。

- **コーディングガイドライン** — カスタムリンターなど
- **組織横断の不変条件** — 認証情報・APIキー・トークンのハードコード禁止など、交渉の余地のないもの
- **ドメイン契約** — フレームワーク・サービス・コードベースの一部に特化（例：決済ドメインではすべての金額は Money 型を使う）
- **受け入れ基準** — タスクに特化したもの

検証ステップは、コードが書かれる**前**に定義すべきであり、既にあるものを確認するために後から作るのではない。エージェントがコードとテストの両方を書くと、問題が「正しいことをテストしているか」に移動するだけになる。検証基準は実装ではなく、**spec から**来る必要がある。

### Layer 3: 人間が受け入れ基準を定義する

人間の価値が発揮されるのは、成功がどう見えるかを上流で定義することである。

ここで Behavior-Driven Development（BDD）が新たに重要になる。自然言語で期待される振る舞いを記述し、その spec をテストとして自動化する、という考え方は元々良かったが、コードも書くなら spec を書くのは追加の仕事に感じられ、広まらなかった。

エージェント時代には式が逆転する。Spec は追加作業ではなく、**主たる成果物**になる。人間が spec を書き、エージェントが実装し、BDDフレームワークが検証する。何かが失敗しない限り、実装を読む必要はない。

人間が受け入れ基準を定義し、機械が検証する。それが本当に意味のあるゲートである。

### Layer 4: 権限システムとしてのアーキテクチャ

このエージェントは何に触れてよいか。何がエスカレーションを要するか。これらは後付けではなく、アーキテクチャの決定になる。

多くのエージェントフレームワークでは権限が all-or-nothing である。シェルアクセスがあるかないか。しかし粒度が重要である。ユーティリティ関数のバグ修正をするエージェントは、インフラ設定へのアクセスは不要。テストを書くエージェントはCIパイプラインを変更する必要はない。

スコープは、エージェントが有用な仕事をするのに必要な範囲で可能な限り狭くする。タスクが「utils/dates.py の日付パースバグを直す」なら、ファイルシステムアクセスはそのファイルとそのテストファイルに限定する。コードベース全体でも「src/ と tests/」でもなく、そのタスクに関係するファイルだけにする。

エスカレーショントリガーも同様に重要である。認証ロジックに触れる、DBスキーマを変更する、新規依存を追加する、といったパターンは、エージェントの自信度に関係なく、自動で人間レビューに回すべきである。

### Layer 5: 敵対的検証

責務の分離：1体が仕事をし、もう1体が検証する。互いを信頼しない。それが目的である。

古いパターンである——QAチームがエンジニアリングマネージャーに報告しない方がよい理由、コードを書く人が唯一のレビュアーであってはいけない理由と同じである。

エージェントでは、これをアーキテクチャで強制できる。コーディングするエージェントは、検証エージェントが何をチェックするか知らない。検証エージェントは、自分の仕事を楽にするためにコードを変更する能力を持たない。設計として敵対的である。

さらに、第三のエージェントが最初のエージェントが作ったものを壊そうと試み、エッジケースや故障モードを狙う、という構成もできる。レッドチーム・ブルーチームを、自動化し、すべての変更で実行する。

---

## 5. ルールとしての整理（記事内の主張）

- Code **must not be written** by humans.
- Code **must not be reviewed** by humans.

人間のチェックポイントは「正しく書いたか？」から「正しい問題を、正しい制約で解いているか？」に移し、最初の1行が生成される前に判断を入れる。

---

## 6. 結論：「良いコード」の意味が変わる

エージェントシステムのインセンティブは単純である：タスクを与えられたら、完了できるか。依頼者を満足させられるか。エージェントの成功は、長期的な正確さやビジネス要件に本来駆動されない。制約にそれを組み込むのは私たちの仕事である。

エージェントが生成し、エージェントが読むコードにおいて、「良いコード」の見え方はより標準化されていく。新規コードベースでは、デフォルトがより一貫するため、指示は少なくて済む。

これからは、**速く出し、すべてを観測し、もっと速く戻す**。遅くレビューし、バグを見逃し、本番でデバッグする、ではない。

機械より多く読むことはできない。上流で、意思決定が本当に起きているところで、機械より深く考える必要がある。

結局、エージェントがコードを問題なく扱えるなら、私たちがそれを読めるかどうかは、どうでもよいのである。
