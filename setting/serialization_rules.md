# 連載執筆ルール

> **参照ファイル**:
> - `config.md`: 書籍の企画概要と章立て
> - `writing_rules.md`: 執筆ガイドライン全般
> - `editor_perspective.md`: 編集者視点の最終確認手順
> - 各章の `chapter_X/chapter_X.md`: 連載各回の素材となる解説パート
> - 各章の `chapter_X/chapter_X_pain_mapping.md`: ペインとシーンの対応関係

## 連載の概要

- 連載タイトル：誤解だらけの開発生産性〜DMM.comで見てきた誤解〜
- 1回あたりの文字数：4000〜5000字（目安4000〜4800字）
- 全6回構成
- 母体は本書『わたしたちは開発生産性を誤解していた。』。各回は本書の対応章を再構成した解決記事。

## 各回のタイトルと対応章

| 回 | タイトル | 主な対応章 |
|---|---|---|
| 第1回 | スピードの誤解〜「速く出せば生産性は上がる」は本当か〜 | 第1章 |
| 第2回 | 数値の誤解〜開発生産性は定量化できるのか〜 | 第2章 + 第4章 |
| 第3回 | 技術的負債の誤解〜「測れないから説明できない」を越える〜 | 第3章 |
| 第4回 | 人月の誤解〜「人が足りない」「AIで増やせる」の幻想〜 | 第5章 |
| 第5回 | 評価の誤解〜エンジニアだけが生産性を求められる構造〜 | 第6章 |
| 第6回 | 標準化の誤解〜トップダウンの全社統一指標は機能するか〜 | 第7章 |

## 全体方針

- 解決記事フォーマット（ペイン提示 → 誤解の解剖 → 解決の方向 → 締め）で書く。
- 物語仕立てにはしない。湊・飛鳥・山田・佐藤・高橋などの登場人物名や、具体的な会社設定（中小企業向けSaaS、勤怠管理モジュール等の固有名）はそのまま使わない。「ある開発現場では」「リーダーが」のように現場ルポとして抽象化する。
- 対応章のストーリーパートで描かれたペインを「現場で起きていること」として要約し、解説パートを軸に組み立てる。
- 文体は書籍の解説パートに準拠した「です・ます調」。
- 必ず日本語UTF-8でファイルを出力する。書き出し後にエンコーディングを確認する（後述「出力ルール」）。
- 連載タイトルが「DMM.comで見てきた誤解」のため、リードまたは本文で「DMM.comの現場で見てきた」というトーンを継承する。

## ファイル構成と命名規則

連載の原稿は `serialization/` 配下に格納する。

```
serialization/
├── 01_speed.md          ← 第1回 スピードの誤解
├── 02_metrics.md        ← 第2回 数値の誤解
├── 03_tech_debt.md      ← 第3回 技術的負債の誤解
├── 04_man_month.md      ← 第4回 人月の誤解
├── 05_evaluation.md     ← 第5回 評価の誤解
└── 06_standardization.md ← 第6回 標準化の誤解
```

命名規則：`{連番02桁}_{英語キーワード}.md`

## 各回の標準構成

| セクション | 文字数目安 | 役割 |
|---|---|---|
| リード | 200〜300字 | 重圧の入口、その回の誤解の提示、連載のトーン継承 |
| 現場で起きていること | 700〜900字 | ストーリーパートのペインを抽象化して現場ルポとして提示 |
| 誤解の解剖 | 1100〜1500字 | 解説パートの中核を再構成。なぜそれが誤解なのかを論理で組み立てる |
| 誤解の延長 | 300〜700字 | 関連する誤解、次回への布石（任意で次回連載へのリンク） |
| 解決の方向 | 900〜1100字 | 実践的アドバイス、明日からできること |
| おわりに | 150〜300字 | 主張の再確認、次回予告 |
| 参考文献 | - | その回で言及した文献のみ。`setting/reference/merged_references.md` §8の表記に合わせる |

合計：4000〜4800字を目安。5000字を大きく超えるなら「誤解の解剖」と「解決の方向」から圧縮する。

## 出力ルール（UTF-8運用）

本プロジェクト環境では、Cursorの `Write` ツールおよび `StrReplace` ツールが日本語をASCIIの `?` に変換してしまう問題が確認されている。再発防止のため、以下の手順で書き込みを行う。

### 新規mdファイルの作成

シェルのヒアドキュメントを使う。クォート付きの `'EOF'` で囲み、変数展開を避ける。

```bash
cat > /path/to/file.md <<'EOF'
（日本語本文）
EOF
```

### 既存mdファイルの部分編集

Pythonでread → 文字列置換 → UTF-8で書き戻し。`StrReplace` ツールは使わない。

```bash
python3 <<'PYEOF'
with open('/path/to/file.md', encoding='utf-8') as f:
    text = f.read()
text = text.replace('（旧）', '（新）')
with open('/path/to/file.md', 'w', encoding='utf-8') as f:
    f.write(text)
PYEOF
```

### 書き出し後の確認

書き出し後に必ず以下を確認する。

- `file path/to/file.md` の出力が `Unicode text, UTF-8 text` になっていること。
- `head -c 60 path/to/file.md | xxd` の先頭バイトがマルチバイトUTF-8（例：`e7ac ac` など）になっていること。`3f` が連続している場合は化けているので削除して再作成する。

### 文字数の数え方

```bash
python3 -c "
import re
with open('/path/to/file.md', encoding='utf-8') as f:
    text = f.read()
# 空白除外で全文をカウント
print('全文（空白除く）:', len(re.sub(r'\s', '', text)))
# セクション別
for sec in re.split(r'^## ', text, flags=re.M):
    title = sec.split('\n')[0] or '冒頭'
    print(f'  {title}: {len(re.sub(chr(0x73)+chr(0x73)+chr(0x73), \"\", sec))}字')
"
```

簡易的には、本文（タイトル・参考文献を除く）の空白除外文字数で判定する。

## 表記ルール（writing_rules.md準拠）

setting/writing_rules.md と setting/editor_perspective.md に全面的に従う。連載で特に外せないものを抜粋する。

### 文体・記号

- 解説記事として全体を「です・ます調」で統一する。
- 読点は最小限。短い修飾語の後の読点は削る（「確かに、」→「確かに」）。並列の「A、B、C」は維持。
- 通常の文章の文末に「：」は使わない。箇条書きの太字見出し直後の「：」のみ可。
- 略語の表記：プルリクエスト（PR）、プロジェクトマネージャー（PM）は初出で正式名称併記、以降は略語のみ。QA（品質保証）は初出で略語＋正式名称、以降はQA。
- 全角括弧（）を使う。半角括弧 () は英文の年号など限定。
- スラッシュ「／」で概念を並列しない。矢印「→」も控える。

### 解説パートとしての書き方

- ストーリーで描かれた具体エピソード（湊・飛鳥・山田・佐藤・高橋などの固有名詞、社内チャンネル名）は使わず、現象だけを抜き出す。
- 主人公の内面描写や心情は入れず、客観的なトーンで書く。
- 「ある開発現場では」「リーダーが」のように一般化する。
- ストーリー外の用語（「第〇章」「手法ガイド」など書籍内構造を指すもの）は本文で使わない。連載構造の参照（「連載第N回」）は許容。

### AIっぽさの排除（writing_rules.md §10）

- 前置き宣言禁止：「結論から言うと」「本記事では」「以下で解説します」など。
- 安全クッション禁止：「一般的に」「多くの場合」「一概には言えませんが」「状況によって異なります」など。
- 抽象語の連呼禁止：「重要」「効果的」「最適」「本質」「メリット」を連呼しない。
- 同義語連打禁止：「重要・大切・欠かせない」のような言い換え連打はしない。
- 抽象まとめ禁止：「まとめると」「要するに」「総じて」など。
- 文末リズムを均一にしない。短文と長文を混ぜる。
- 締めの定型句禁止：「参考になれば幸いです」「まずは小さく始めましょう」など。
- 「」の多用禁止。定義や強調目的のカギ括弧は文脈に溶かす。引用や固有の呼称のみに残す。
- () の多用禁止。補足は本文の一文に組み込む。
- Markdown装飾：太字（`**…**`）は強調が機能する箇所のみに限定（各セクションの小見出し相当）。

## 自己チェック手順（執筆後の必須項目）

執筆完了直後に必ず以下を実行する。問題が見つかったら修正して再チェックする。

1. **エンコーディング確認**
   - `file serialization/0X_xxx.md` が `Unicode text, UTF-8 text` か。
2. **文字数チェック**
   - 合計が4000〜4800字に収まっているか。
   - 各セクションが目安の幅にだいたい収まっているか（±20%程度）。
3. **表記ルール違反チェック**
   - 前置き宣言、安全クッション、締めの定型句、抽象語連呼、スラッシュ、矢印、半角括弧の濫用がないか。
   - 略語の初出表記、全角括弧の使用が適切か。
4. **編集者視点5観点**（setting/editor_perspective.md準拠）
   - 構造分析：主張は一本化されているか。各段落の役割は明確か。論理に飛躍はないか。
   - 論理展開：因果が明確か。同義反復はないか。接続詞が不自然でないか。
   - 冗長表現：削っても意味が変わらない箇所、修飾過多、曖昧語の連発はないか。
   - 読者視点：「ではどうすれば」に答えているか。離脱しそうな密度はないか。具体化不足はないか。
   - 改稿の必要性：あれば理由とセットで該当箇所を改稿する。

## 各回の執筆手順（標準フロー）

新しい回を書くときは、以下の順序で進める。

1. このファイル（setting/serialization_rules.md）を読み込む。
2. setting/writing_rules.md と setting/editor_perspective.md を読み込む。
3. 該当章の chapter_X/chapter_X.md と chapter_X/chapter_X_pain_mapping.md を読み込む。
4. 必要に応じて setting/config.md、setting/pain_points.md、setting/reference/merged_references.md を参照する。
5. プランを作成し、ユーザーに確認を取る（章のどの解説パートを軸にするか、どのフレームワーク・参考文献を引用するか、セクション分量配分）。
6. 承認後、シェルヒアドキュメントで原稿を新規作成する。
7. 自己チェック手順を実行する。
8. 必要に応じてPython経由で部分編集する。
9. ユーザーに報告する。
